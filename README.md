# Laravel Actions and Services

![Laravel Actions and Services](assets/poster.jpg)

Laravel gives you a lot of freedom about where business logic goes. After a year of "nobody stops you," the order flow lives in four places and you are afraid to touch any of them.

Two layers fix this: Actions for complete operations, Services for focused reusable work. An opinionated guideline based on real-world Laravel projects, not abstract theory.

## What's Inside

- The one rule that decides Action vs Service: direction, not purity
- Quick-check questions to place any class in under a minute
- Why Services must never call Actions or dispatch jobs (the hard invariant)
- How to compose Actions and wire them through controllers
- Repositories, DTOs, and exceptions in this layering model
- When not to bother - because simple CRUD does not need this
